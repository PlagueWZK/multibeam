# 细网格驱动的点取舍与测线停止需求文档

**文档ID**: REQ-2026-04-11-001  
**创建时间**: 2026-04-11  
**状态**: Frozen

---

## 1. Goal (目标)

将当前测线规划从“环形检测 + 收敛状态 + 饱和算法”的事后终止模式，切换为“细网格状态驱动的点取舍与测线停止”模式。

---

## 2. Deliverable (交付物)

- `Planner.py` 中主测线/从测线生成逻辑改造
- `coverage_state_grid.py` 中可复用的候选点/候选线段收益判断接口
- 新的停止口径：按当前测线规划类型决定是否结束当前线或当前方向扩展
- 最少一轮静态验证，外加一轮运行级或输出级验证

---

## 3. Constraints (约束条件)

1. 保持单一运行时权威，不新增第二套路由或第二套规划主流程。
2. 保留边界检测作为硬约束。
3. 去除主流程对以下逻辑的依赖：
   - 环形/自交检测
   - 历史测线相交检测
   - 收敛状态判断
   - 饱和算法
4. 点是否保留，必须由细网格当前状态决定，而不是由几何闭合趋势决定。
5. 停止逻辑必须区分当前规划阶段：
   - 主测线双向延伸
   - 从测线平移生成下一轮候选线
   - 从测线两端自延伸

---

## 4. Acceptance Criteria (验收标准)

- [ ] `Planner.py` 不再调用环形检测、收敛状态、饱和算法作为终止依据
- [ ] 主测线新点会先经过细网格收益判断，再决定是否加入
- [ ] 从测线平移生成时，每个候选点会先经过细网格收益判断
- [ ] 从测线两端自延伸时，新点会先经过细网格收益判断
- [ ] 主测线在两端都不可继续时停止
- [ ] 从测线方向在“下一轮有效点不足”时停止
- [ ] 输出仍可完成细网格覆盖状态统计与最终绘图

---

## 5. Product Acceptance Criteria (产品验收标准)

- 测线规划应更偏向覆盖增益驱动，而不是环形后处理驱动
- 重复覆盖严重区域应更早被拒绝，而不是等形成闭合后再终止
- 规划结果仍保持当前分区/双网格主线兼容

---

## 6. Manual Spot Checks (手动验证点)

1. 检查 `Planner.py` 中已无 `check_self_intersection` / `check_line_intersection_shapely` 驱动终止
2. 检查主测线和从测线扩展都存在“细网格收益不足即停止/拒绝”的分支
3. 运行一次规划，确认程序能输出最终结果而非因新逻辑崩溃
4. 观察日志，确认停止原因主要变为边界或细网格低收益，而非螺旋/饱和

## 11. Inferred Assumptions (推断假设)

1. “根据细网格状态判断该点取舍”在本轮放宽为：候选点/候选线段只要触达任何当前 `non-full` 细网格 cell，就不视为低收益；仅当其作用范围内全部为 `full` cell 时才拒绝。
2. “根据当前测线规划类型判断是否停止”默认解释为：
   - 主测线：两端都因边界或低收益不可继续时停止
   - 从测线平移生成：经筛选后的有效点数 `<= 1` 时停止该方向
   - 从测线自延伸：两端都因边界或低收益不可继续时停止当前线
3. 当前任务优先追求停止逻辑统一和稳定，不要求保留旧的 `spiral / saturation` 可视化语义。

---

## 12. Frozen Decision Summary (冻结决策摘要)

| 决策面 | 冻结结果 |
|---|---|
| 点取舍依据 | 细网格触达任何 non-full cell 即保留 |
| 主测线停止 | 两端均不可继续 |
| 从测线生成停止 | 下一轮有效点数 `<=1` |
| 从测线自延伸停止 | 两端均不可继续 |
| 旧逻辑状态 | 环形/收敛/饱和退出主流程 |
